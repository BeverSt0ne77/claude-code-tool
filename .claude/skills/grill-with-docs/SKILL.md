## 领域感知

在浏览代码库的过程中，也请查找已有的文档：

### 文件结构

大多数代码库只有一个上下文（context），结构如下：

```
/
├── CONTEXT.md
├── docs/
│   └── adr/
│       ├── 0001-event-sourced-orders.md
│       └── 0002-postgres-for-write-model.md
└── src/
```

如果根目录存在 `CONTEXT-MAP.md`，则表示该仓库包含**多个上下文**。该映射文件会指明每个上下文所在的位置：

```
/
├── CONTEXT-MAP.md
├── docs/
│   └── adr/                          ← 系统级决策
├── src/
│   ├── ordering/
│   │   ├── CONTEXT.md
│   │   └── docs/adr/                 ← 上下文专属决策
│   └── billing/
│       ├── CONTEXT.md
│       └── docs/adr/
```

请**按需创建**文件 —— 只有当你确实有内容需要写入时才创建。如果 `CONTEXT.md` 不存在，则在第一个术语确定时创建它。如果 `docs/adr/` 不存在，则在需要记录第一个 ADR 时创建该目录。

## 工作过程中的要求

### 对照术语表进行质疑

当用户使用的术语与 `CONTEXT.md` 中已有的语言定义发生冲突时，请**立即指出**。例如：“您的术语表中将‘取消’定义为 X，但您刚才似乎是在说 Y —— 究竟是哪一个？”

### 精炼模糊的措辞

当用户使用模糊或含义过载的词汇时，请提出一个精确的标准术语。例如：“您说的‘账户’—— 是指客户（Customer）还是用户（User）？这两者是不同的概念。”

### 讨论具体场景

在讨论领域关系时，请用具体的场景进行“压力测试”。编造一些能触及边界情况的场景，迫使用户在概念边界上做出精确的界定。

### 与代码交叉验证

当用户描述某个功能如何工作时，请检查代码是否真的如此实现。如果发现矛盾，请明确指出：“您的代码是取消整个订单，但您刚才说可以部分取消 —— 哪一个才是正确的？”

### 即时更新 CONTEXT.md

每当一个术语被确定下来，请**立即**更新 `CONTEXT.md`，不要积累到后面统一更新。格式请参考 [CONTEXT-FORMAT.md](./CONTEXT-FORMAT.md)。

`CONTEXT.md` 应当**完全不包含任何实现细节**。请不要将 `CONTEXT.md` 当作规格说明、草稿本或实现决策的仓库 —— 它只应当是术语表，仅此而已。

### 谨慎提供 ADR

**仅当以下三个条件全部满足时**，才提议创建 ADR：

1. **难以逆转** —— 将来改主意成本较高；
2. **没有上下文则令人费解** —— 未来的阅读者会困惑“为什么要这样做？”；
3. **源自真实权衡** —— 确实存在可行的替代方案，并且你基于具体原因选择了当前方案。

如果三者中有任意一点不满足，就跳过 ADR。ADR 的格式请参考 [ADR-FORMAT.md](./ADR-FORMAT.md)。
