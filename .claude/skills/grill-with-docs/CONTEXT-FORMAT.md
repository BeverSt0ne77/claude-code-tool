# CONTEXT.md 格式说明

## 结构

```md
# {上下文名称}

{用一两句话描述这个上下文是什么以及它为什么存在。}

## 术语表

**Order（订单）**：
{用一两句话描述该术语}
_避免使用_：Purchase（购买）、transaction（交易）

**Invoice（发票）**：
发货后发送给客户的付款请求。
_避免使用_：Bill（账单）、payment request（付款请求）

**Customer（客户）**：
下单的个人或组织。
_避免使用_：Client（客户）、buyer（买家）、account（账户）
```

## 规则

- **保持有主见。** 当同一概念存在多个词汇时，选出最好的一个，并把其他词列入 `_避免使用_`。
- **定义要精炼。** 最多一两句话。定义它**是什么**，而不是它做什么。
- **只纳入特定于此项目上下文的术语。** 通用编程概念（超时、错误类型、工具模式）即使项目大量使用，也不应列入。在添加术语之前，先问自己：这是此上下文独有的概念，还是通用编程概念？只有前者才应该收入。
- **当自然聚类出现时，将术语归入子标题下。** 如果所有术语都属于一个紧密相关的领域，那么平铺列表也完全可以。

## 单上下文 vs 多上下文仓库

**单上下文（大多数仓库）：** 仓库根目录下只有一个 `CONTEXT.md`。

**多上下文：** 仓库根目录下有一个 `CONTEXT-MAP.md`，列出各个上下文、它们的位置以及它们之间的关系：

```md
# 上下文映射

## 上下文列表

- [订单上下文](./src/ordering/CONTEXT.md) —— 接收并跟踪客户订单
- [计费上下文](./src/billing/CONTEXT.md) —— 生成发票并处理付款
- [履约上下文](./src/fulfillment/CONTEXT.md) —— 管理仓库拣货和发货

## 上下文关系

- **订单 → 履约**：订单上下文发出 `OrderPlaced` 事件；履约上下文消费这些事件以开始拣货
- **履约 → 计费**：履约上下文发出 `ShipmentDispatched` 事件；计费上下文消费这些事件以生成发票
- **订单 ↔ 计费**：共享 `CustomerId` 和 `Money` 类型
```

本技能会自动推断应使用哪种结构：

- 如果 `CONTEXT-MAP.md` 存在，则读取它以找到各个上下文
- 如果仅根目录下存在 `CONTEXT.md`，则为单上下文
- 如果两者都不存在，则在第一个术语确定时，按需在根目录创建 `CONTEXT.md`

当存在多个上下文时，会推断当前讨论的主题属于哪个上下文。如果不清楚，则会询问你。
