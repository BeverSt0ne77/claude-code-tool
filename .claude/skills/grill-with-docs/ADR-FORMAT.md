# ADR 格式说明

ADR（架构决策记录）存放在 `docs/adr/` 目录下，使用连续编号命名：`0001-slug.md`、`0002-slug.md`，以此类推。

请**按需创建** `docs/adr/` 目录 —— 只有在需要记录第一个 ADR 时才创建它。

## 模板

```md
# {决策的简短标题}

{1-3 句话：背景是什么、我们做了什么决策、以及为什么。}
```

就这么简单。一篇 ADR 可以只是一个段落。它的价值在于记录*做了哪个决策*以及*为什么做*—— 而不是为了填满各个章节。

## 可选章节

仅当这些章节确实能增加价值时才包含它们。大多数 ADR 并不需要。

- **状态** 前置元数据（`proposed | accepted | deprecated | superseded by ADR-NNNN`）—— 当决策被重新审视时很有用
- **考虑过的方案** —— 仅当被拒绝的替代方案值得被记住时
- **后果** —— 仅当需要指出不明显的影响下游的连锁反应时

## 编号规则

扫描 `docs/adr/` 目录，找到当前最大的编号，然后加一。

## 何时应该提议创建 ADR

以下三个条件**必须全部满足**：

1. **难以逆转** —— 将来改变主意的成本较高；
2. **没有上下文则令人费解** —— 未来的阅读者看到代码时会困惑“到底为什么要这样做？”；
3. **源自真实权衡** —— 确实存在可行的替代方案，并且你基于具体原因选择了当前方案。

如果一个决策很容易逆转，那就跳过它 —— 反正你将来也会把它改回去。如果它不令人费解，也不会有人困惑“为什么”。如果根本没有真正的替代方案，那就没什么可记录的，除了“我们做了显而易见的选择”。

### 什么样的决策值得记录

- **架构形态。** “我们采用单体仓库（monorepo）。” “写模型是事件溯源的，读模型被投影到 PostgreSQL 中。”
- **上下文之间的集成模式。** “订单上下文和计费上下文通过领域事件通信，而不是同步 HTTP 调用。”
- **带有锁定效应的技术选型。** 数据库、消息总线、认证提供商、部署目标。不是每个库都要记 —— 只要那些换掉需要花一个季度的。
- **边界和范围决策。** “客户数据归属于客户上下文；其他上下文仅通过 ID 引用它。” 明确说出的“不做什么”和“做什么”同样有价值。
- **偏离常规路径的有意选择。** “我们使用手写 SQL 而不是 ORM，因为 X。” 任何理性读者会默认认为相反做法的地方。这些记录能阻止下一个工程师“修复”某个其实是故意的设计。
- **代码中不可见的约束。** “由于合规要求，我们不能使用 AWS。” “由于合作方 API 合同的约定，响应时间必须低于 200ms。”
- **被否决的替代方案，且否决理由不明显。** 如果你曾考虑过 GraphQL 但出于微妙原因选择了 REST，那就记录下来 —— 否则六个月后又会有人再次建议使用 GraphQL。
